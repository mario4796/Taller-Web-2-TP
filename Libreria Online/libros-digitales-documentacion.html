Libros Digitales — Documentación técnica

Proyecto: Librería Online · Taller Web II · Fecha: 27/06/2026

1. Base de datos (PostgreSQL + Prisma)

Dos tablas son responsables de toda la lógica de libros digitales.

Tabla Libros

Tiene el campo archivoDigital VARCHAR(500) que guarda únicamente el nombre del archivo PDF (por ejemplo: TomSawyer-Twain.pdf). No guarda el PDF dentro de la base de datos, solo el nombre para poder ubicarlo en el sistema de archivos del servidor.

Tabla LibroDigitalAdquirido

Registra qué comprador adquirió qué libro. Tiene una restricción UNIQUE(comprador_id, libro_id) para evitar adquisiciones duplicadas.

ColumnaTipoDescripción
idSERIALClave primaria autoincremental
comprador_idINTEGERFK → Compradores.usuario_id
libro_idINTEGERFK → Libros.id
fecha_adquisicionTIMESTAMPSe asigna automáticamente al momento de la compra

El ORM utilizado es Prisma, que genera un cliente TypeScript a partir del esquema definido en back/prisma/schema.prisma. Para que el backend reconozca una nueva tabla, es necesario ejecutar npx prisma generate luego de cada cambio en el esquema.

2. Backend (Express + TypeScript + Bun)

El backend expone tres endpoints bajo la ruta base /api/libros-digitales, definida en back/src/routes/routes.ts y manejada por libro-digital.router.ts.

MétodoRutaQué hace
POST /adquirir Registra una nueva adquisición en LibroDigitalAdquirido
GET /comprador/:id Devuelve todos los libros adquiridos por un comprador
GET /pdf/:libroId?compradorId=X Verifica la posesión y sirve el archivo PDF al navegador

Validaciones en libro-digital.service.ts

Antes de registrar una adquisición, el servicio valida tres condiciones en orden:

// 1. El libro existe en la base de datos
if (!libro) throw new Error('Libro no encontrado');

// 2. El libro tiene un archivo digital asignado
if (!libro.archivoDigital) throw new Error('El libro no tiene versión digital disponible');

// 3. El comprador no lo adquirió antes (UNIQUE constraint)
if (yaAdquirido) throw new Error('El comprador ya posee este libro digital');

Entrega del PDF

El endpoint GET /pdf/:libroId usa res.sendFile() de Express para enviar el archivo directamente desde el sistema de archivos al navegador: 

const rutaArchivo = path.resolve('uploads', nombreArchivo);
res.sendFile(rutaArchivo);

Los PDFs se almacenan en la carpeta back/uploads/ del servidor. El nombre del archivo en esa carpeta debe coincidir exactamente con el valor del campo archivoDigital en la base de datos.

3. Frontend (Angular 17 + PrimeNG + Signals)

Mostrar el botón "Versión digital"

El componente libro-estanteria recibe cada libro como un signal de entrada (input.required<Libro>). La interfaz Libro en shared/interfaces/libro.interface.ts incluye archivoDigital?: string | null.

El mapper libro.mapper.ts copia el campo desde la respuesta de la API. En el template, el botón se muestra condicionalmente: 

@if (libro().archivoDigital) {
  <p-button label="Versión digital" icon="pi pi-file-pdf"
    (onClick)="emitirCompraDigital()" />
}

Si el campo es null o undefined, el botón directamente no se renderiza. No hay ningún estado de visibilidad oculto.

Flujo de adquisición

Click en "Versión digital" (libro-estanteria)
→ emitirCompraDigital() verifica que haya sesión activa
→ output onComprarDigital emite el libroId
→ manejarCompraDigital(libroId) en estanteria.ts
→ LibroDigitalService.adquirirLibro(compradorId, libroId)
→ POST /api/libros-digitales/adquirir
→ Toast de éxito / advertencia (ya adquirido) / error

Ver "Mis libros digitales"

La ruta /comprador/libros-digitales carga el componente LibrosDigitales. Al inicializar, llama a: 

this.libroDigitalService.getLibrosAdquiridos(this.authService.getUser())

Esto hace GET /api/libros-digitales/comprador/:id. El backend usa Prisma con include: { Libros: true }, que hace un JOIN automático y devuelve cada registro de adquisición junto con todos los datos del libro. Por eso en el template se accede como adquirido.Libros.nombre.

Los libros se muestran en una tabla de PrimeNG (p-table) con paginador de 8 elementos por página.

Leer el PDF

Click en "Leer PDF" (libros-digitales)
→ getPdfUrl(libroId, compradorId) arma la URL
→ window.open(url, '_blank') abre una pestaña nueva
→ GET /api/libros-digitales/pdf/:libroId?compradorId=X
→ Backend verifica que el comprador posea el libro
→ res.sendFile() entrega el PDF
→ El navegador muestra el PDF con su visor nativo

4. Sesión y autenticación

El AuthService guarda la sesión en localStorage. El método getUser() devuelve el id del usuario logueado, que es el mismo valor que Compradores.usuario_id. Por eso ese ID se usa directamente como compradorId en todas las llamadas a la API de libros digitales sin ningún mapeo adicional.

6. Punto frágil a tener en cuenta

Dependencia de ruta física en el servidor El backend sirve los PDFs con path.resolve('uploads', nombreArchivo). Esto significa que el proceso de Node/Bun debe ejecutarse desde el directorio back/ y la carpeta uploads/ debe existir en ese mismo nivel. Si el servidor se despliega en otro entorno o se cambia el directorio de trabajo, hay que garantizar que esa ruta siga siendo válida o actualizar la configuración.

7. Resumen del ciclo completo de un PDF

Archivo físico: back/uploads/TomSawyer-Twain.pdf

↓ El nombre se guarda en la BD:
  Libros.archivoDigital = 'TomSawyer-Twain.pdf'

↓ El frontend recibe ese campo y muestra el botón "Versión digital"

↓ Al adquirir, se crea un registro en LibroDigitalAdquirido

↓ Desde "Mis libros digitales", click en "Leer PDF":
  GET /api/libros-digitales/pdf/2?compradorId=3

↓ Backend verifica posesión → res.sendFile()

↓ El navegador muestra el PDF en una pestaña nueva
